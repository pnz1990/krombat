# Kubernetes RPG — Design Document

## Overview
A turn-based dungeon RPG where the entire game state lives in Kubernetes, orchestrated by 9 kro ResourceGraphDefinitions on Amazon EKS. No custom controllers, no external databases. Just declarative resource graphs, CEL expressions, and kro turning Kubernetes into a programmable game engine.

## Architecture

### Resource Model
- **Dungeon CR** (parent) — owns all game state in spec fields
- **Hero CR** → ConfigMap (HP, class, mana, stats via CEL)
- **Monster CR** → ConfigMap (alive/dead state derived from HP via CEL)
- **Boss CR** → ConfigMap (pending/ready/defeated from HP + monstersAlive via CEL)
- **Treasure CR** → ConfigMap + Secret (opened/unopened state via CEL)
- **Modifier CR** → ConfigMap (curse/blessing effects via CEL)
- **Loot CR** → Secret (item data: type, rarity, stat via CEL)
- **Attack CR** → Job (combat only: monster/boss attacks, abilities)
- **Action CR** → Job (non-combat: equip, use item, treasure, door, room transition)

### State Flow
1. User creates Dungeon CR via backend API
2. kro's dungeon-graph reconciles: creates Namespace, Hero, Monster, Boss, Treasure, Modifier CRs
3. Each child RGD reconciles its CR into native K8s resources (ConfigMaps, Secrets)
4. User submits Attack CR (combat) or Action CR (items) via backend API
5. kro's attack-graph/action-graph reconciles: creates a Job
6. Job reads Dungeon CR, computes result, patches Dungeon CR spec
7. kro re-reconciles child CRs based on updated Dungeon spec
8. Frontend polls Dungeon CR for state changes

### Multi-Room Dungeons
- Room 1: goblins/skeletons + dragon boss
- After boss kill: treasure auto-opens → door auto-unlocks → click door to enter Room 2
- Room 2: trolls/ghouls + bat-boss (harder HP values)
- Room 2 is final — no more treasure/door after Room 2 boss

### Combat System
- D&D-style dice: Easy 1d20+2, Normal 2d12+4, Hard 3d20+5
- All combat logic in Attack Job bash scripts
- attackSeq counter on Dungeon CR for reliable poll detection
- Loot drops only on kill transition (OLD_HP > 0 && NEW_HP == 0)
- Status effects: Poison (-5/turn), Burn (-8/turn), Stun (skip attack)
- Dungeon modifiers: 6 types (3 curses, 3 blessings)

### Item System
- Items handled by Action CR → action-graph RGD (separate from combat)
- Weapons: +damage for 3 attacks
- Armor: +defense (reduce counter damage)
- Shields: block chance (negate counter)
- HP/Mana potions: instant restore
- Items don't cost a turn (no counter-attack)

### Key Design Decisions
- CRs as the only interface — backend/frontend ONLY touch Dungeon, Attack, Action CRs
- Frontend derives gameOver/bossState from spec fields (not status — status can be stale after room transitions)
- Attack Jobs increment attackSeq for reliable frontend polling
- "Already dead" attacks don't increment attackSeq (prevents false positives)
- action-graph handles all non-combat to prevent loot/equip bugs in attack-graph
- Boss target matching uses "-boss$" suffix (not just "boss" anywhere in name)

## Infrastructure
- EKS Auto Mode cluster (K8s 1.34) in us-west-2
- kro + Argo CD as EKS Managed Capabilities
- Custom job-runner image (alpine + kubectl + jq + bash, ~30MB)
- DaemonSet caches job-runner image on all nodes
- Node-warmer with anti-affinity ensures spare node capacity
- CI: GitHub Actions builds images → ECR → rollout restart
- Access: kubectl port-forward svc/rpg-frontend -n rpg-system 3000:3000
